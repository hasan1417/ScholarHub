A researcher's day is
stitched across 4–5 tools
that don't talk to each other.

Each tool optimizes its own narrow concern — none is aware of the others. The 2024 Springer chapter From Fragmentation to Focus calls this the dominant productivity bottleneck in modern scholarly work.

The hidden cost

Walters & Wilder (2023): 55% of citations produced by GPT-3.5 and 18% by GPT-4 were entirely fabricated. Subsequent work measured 78–90% hallucination in GPT-4o without retrieval grounding.

Project Main Idea

One project graph.
Three workflows.
Shared context throughout.

A researcher creates a project. Inside it: a paper library, discussion channels, multi-file LaTeX documents, meetings — all visible to the same AI assistant. Every answer, edit, and search is grounded in your project's actual data.

The integration is the product, not any single feature.05 / 31
Project Contributions06 / 31

Six contributions of this work.

C1 · ARCHITECTURE Unified architecture for grounded research writing. Discovery, references, real-time LaTeX, and library-grounded AI in a single deployable system.
C2 · AI AGENT Tool-orchestrated AI agent grounded in a curated library. 30 typed tools. A post-generation filter drops any citation key not present in the project library.
C3 · DISCOVERY Multi-source paper discovery with deduplication. Nine open APIs aggregated in parallel; union-find dedup with three-tier preference scoring.
C4 · EDITOR Multi-file CRDT editor for LaTeX. Per-file Y.Text binding with independent undo managers — extends Y.js beyond the single-document use case.
C5 · EVALUATION Empirical evaluation across four 2026 frontier LLMs. Ungrounded vs grounded benchmark with automated citation verification. Open and reproducible.
C6 · ROADMAP A productization roadmap. Concrete architectural changes for multi-tenancy, sandboxed AI compute, and institutional deployment.

Citation Hallucination10 / 31

LLMs fabricate citations
— consistently and at scale.

A fabricated citation looks bibliographically real. The author, year, and title are syntactically plausible. The paper does not exist. This is a documented academic integrity concern.

Retrieval-Augmented Generation11 / 31

Ground the model
in real documents.

Lewis et al.\ (NeurIPS 2020) introduced RAG: retrieve relevant documents from an indexed corpus, then condition generation on them. The output cites what it retrieved — not what it imagined.

01
Vector retrieval

Dense embeddings of paper abstracts, indexed by pgvector inside Postgres.

02
Context injection

Library entries with citation keys placed in the system prompt at generation time.

03
Empirically validated

OpenScholar (2025) reaches expert-level citation accuracy across 45M papers.

Why this matters here

RAG alone is not enough — the model can still invent citation keys that look right.
ScholarHub adds a post-filter that drops any key not in the project library.

Grounded + filtered → zero fabricated keys by construction.
Lewis et al. 2020 · Asai et al. 202511 / 31
Conflict-Free Replicated Data Types12 / 31

Real-time co-editing,
without merge conflicts.

Shapiro et al.\ (2011) formalized data types whose concurrent edits commute mathematically. Two users can edit the same line at the same time — the merge is provably the same on both sides, regardless of message ordering.

SHAPIRO 2011CRDT formalizationEventual consistency without a serializing authority.
Y.JSProduction CRDT libraryOptimized for text. Used by JupyterLab, Atlassian, Linear.
AUTOMERGEJSON CRDTsKleppmann et al. — full structured-data CRDT semantics.

Operational Transformation (Google Docs, Etherpad) achieves the same goal differently — it transforms operations against history. CRDTs were chosen here for their simpler reasoning model and offline-edit support.

Shapiro 2011 · Y.js · Kleppmann 201712 / 31
Tool-Orchestrated Agents13 / 31

Agents that call functions,
not just emit text.

A modern LLM agent doesn't have to guess. It can call typed tools — search the library, fetch a paper, create a citation — and ground its output on real return values.

01
Function calling protocols

OpenAI, Anthropic, Gemini all converged on tool/function-call schemas in 2023–2024.

02
ReAct & tool-use loops

The agent decides what to call, observes the result, decides again — until done.

03
Project-scoped tools

Tools in ScholarHub are typed to the user's project — not the open web.

Why tools, not text-only?

When the agent needs to cite paper X, it doesn't recall — it calls get_library_paper(id).
The return value is the canonical reference, every time.

Tools turn LLMs from confident generators into reliable orchestrators.
Function-calling has matured into the dominant agent design pattern.13 / 31
The Gap14 / 31

The building blocks exist.
The integration does not.

Each of the four foundations — tools, RAG, CRDTs, agents — is mature in isolation. Open research workspaces that combine all four do not exist as of 2026.

FOUNDATION 01Open scientific APIsSemantic Scholar, OpenAlex, arXiv, CrossRef, PubMed, CORE — structured metadata for hundreds of millions of papers, free.
FOUNDATION 02RAG + post-filteringReduces fabrication; with a library-key filter, eliminates fabricated citations from the final response.
FOUNDATION 03Multi-file CRDTsY.js + Hocuspocus stack supports concurrent edits on hundreds of files per project.
FOUNDATION 04Tool-orchestrated agentsFunction-calling LLMs reach for typed, project-scoped tools instead of generating from memory.
ScholarHub's contribution is to compose these — not invent them.14 / 31

Citation-Faithful Generation21 / 31

Two layers, zero fabrications.

The model is grounded with the user's library at prompt time. A second filter validates every citation key against the library at response time. Anything that doesn't resolve is dropped before the user sees it.

LAYER 01 · PROMPTING
Library injected into context.
Citation keys, titles, authors, DOIs of relevant papers are placed in the system prompt. The model is instructed to cite only from this list.
LAYER 02 · POST-FILTER
Library-key validation on output.
Every \cite{key} in the response is checked against the project library. Non-matching keys are dropped — the model cannot leak a fabricated citation to the user.
Effect on user-visible citations:
0% fabricated by construction
Empirical numbers on slide 24.21 / 31

What I Learned29 / 31

Engineering lessons.

SYSTEMS DESIGN A typed shared graph is worth more than any single feature. The compounding value of ScholarHub is that library, AI, and editor all read from the same source of truth.
AI ENGINEERING RAG is necessary but not sufficient. A post-generation filter does the heavy lifting on faithfulness — grounding alone leaves residual fabrication.
REAL-TIME CRDTs simplify the mental model. Multi-file Yjs over Hocuspocus is more boring — in the good way — than I expected. The math does its job.
EVALUATION Systems evaluation is rigorous when honest. Empirical benchmarks + measured latencies + action accounting answer the integration question without a full user study.

What I Learned30 / 31

Process & product lessons.

SCOPE DISCIPLINE Defending scope is a daily activity. An integration project has infinite feature-additions. Saying "future work" out loud kept the deliverable real.
SHIPPING Deployment changes how you build. Running scholarhub.space publicly surfaced architecture issues that local development never would.
DOCUMENTATION Write the report alongside the code. The architecture chapter improved the architecture. Explaining a design exposes its weak parts.
FRONTIER MODELS Models will change — the workspace should not. Model-agnostic orchestration keeps ScholarHub durable as the underlying LLM landscape shifts.
